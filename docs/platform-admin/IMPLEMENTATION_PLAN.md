# خطة تنفيذ لوحة الإدارة والصلاحيات المركزية

## قواعد التنفيذ

- كل مرحلة مستقلة وقابلة للمراجعة والتراجع.
- لا يبدأ SQL قبل اعتماد Architecture وERD ومصفوفة الصلاحيات.
- كل Schema change في Migration مستقلة، تطبق أولًا في بيئة اختبار.
- RLS والاختبارات السلبية جزء من المرحلة وليست عملًا لاحقًا.
- لا ينتقل أي موديل إلى القرار المركزي دون Compatibility Mode.
- القرارات الفنية الآمنة في وثائق الإدارة المركزية معتمدة كأساس؛ لا يعاد فتحها في كل مرحلة إلا إذا ظهر تعارض أمني أو تشغيلي موثق.
- ينشأ حساب `system_owner` جديد ومستقل، ولا يغير التنفيذ صلاحيات الحساب التشغيلي الحالي.

## المرحلة 1: التوثيق واعتماد ERD

- **الهدف:** اعتماد المفاهيم والجداول والأدوار والنطاقات والقرارات المفتوحة.
- **الجداول:** لا يوجد.
- **الشاشات:** Wireframes فقط.
- **المخاطر:** غموض دور `platform_admin` وMapping الجهات والأدوار القديمة.
- **شروط القبول:** توقيع مالك المشروع على الوثائق، قاموس المصطلحات، وقرارات ERD المفتوحة.
- **أثر الإنتاج:** لا يوجد.

## المرحلة 2: Module Registry

- **الهدف:** إنشاء `platform_modules` كطبقة حوكمة `draft/active/disabled` مع إبقاء `workspaces` أساسًا تشغيليًا.
- **الحالة:** طبقت Schema المرحلة الأولى على Supabase في Migration `20260715111453_platform_modules_schema`، والجدول موجود وفارغ دون إدخال موديلات.
- **الجداول:** `platform_modules`، مع استخدام `audit_logs` الحالي مباشرة لتاريخ التغيير دون جدول تدقيق مكرر في هذه المرحلة.
- **الشاشات:** قائمة الموديلات وتفاصيل قراءة فقط أولًا.
- **المخاطر:** تعارض حالة Workspace القديمة مع حالة الموديل.
- **نتيجة الاختبار:** نجح إنشاء سجل `draft`، ومنع تكرار `module_code` و`route_path` وربط Workspace ثانٍ، ومنع تعديل الرمز والحذف والكتابة المباشرة، وزاد `lock_version` عند التعطيل. استخدمت سجلات `financial-control` و`spending-efficiency` اختبارياً ثم أزيلت بالـRollback.
- **شروط القبول المتبقية قبل التطبيق التشغيلي:** إنشاء `platform_role_assignments` وRPCs الإدارية لاحقًا كي يرى `system_owner` الموديلات المعطلة؛ لا ينفذ ذلك في هذه المرحلة، ولا يتغير Router الحالي.
- **أثر الإنتاج:** Schema إضافية غير نافذة؛ لم يتغير Router أو بيانات الموديلات الحالية.

## المرحلة 3: Organizations

- **الهدف:** إنشاء `organizations` كهيكل جهات هرمي مع Mapping انتقالي آمن إلى `departments`.
- **الحالة:** طبقت Schema المرحلة الأولى على Supabase في Migration `20260715111519_organizations_schema`، والجدولان موجودان وفارغان دون إدخال جهات أو روابط.
- **الجداول:** `organizations` و`organization_department_mappings` فقط. يؤجل `user_organizations` لمرحلة الصلاحيات المركزية.
- **الشاشات:** شجرة الجهات وتفاصيلها وMapping مراجعة.
- **المخاطر:** حلقات الهيكل وMapping خاطئ لـ`departments`.
- **نتيجة الاختبار:** نجح إنشاء أصل وفرع، ومنع الأب الذاتي والدورة وتكرار الرمز والربط، والسماح بأكثر من قسم للجهة، ومنع أكثر من ربط رئيسي للقسم، ومنع الكتابة المباشرة، وزيادة `lock_version`. أنشئت Departments اختبارية داخل المعاملة فقط لأن الجدول الحي خالٍ، ثم أزيلت بالـRollback.
- **شروط القبول المتبقية:** اعتماد أكواد الجهات الرسمية وRPCs الإدارة وخطة مراجعة Mapping؛ لا Mapping تلقائي ولا تعديل لـ`departments`.
- **أثر الإنتاج:** إضافة موازية؛ `departments` يبقى نافذًا.

## المرحلة 4: Platform Role Assignments

- **الهدف:** إنشاء المصدر المركزي لأدوار `system_owner/platform_admin/auditor/viewer` دون تعديل `profiles.role`.
- **الحالة:** أعدت مسودتا Schema وRPC واختبرتا فقط في معاملات منتهية بـ`ROLLBACK`؛ لم تطبقا ولم ينشأ `system_owner`.
- **الجداول:** `platform_role_assignments` مع `audit_logs` الحالي.
- **الدوال:** Helpers داخل `private`، غلاف قراءة ذاتي آمن، وRPCs المنح والتحديث والإلغاء ومزامنة الحالة.
- **الاختبارات المنجزة:** المالك الوحيد، حدود `platform_admin`، منع self-grant، التواريخ والتداخل، القرار الزمني، `lock_version=40001`، RLS للمستخدم والمدقق، Audit، والتسليم الذري للمالك.
- **Bootstrap:** تصميم داخلي فقط يقبل UUID صريحًا ويُسقط بعد الاستخدام في Migration مستقلة لاحقة.
- **المخاطر:** فقد آخر مالك أو وجود مالكين متزامنين؛ يعالجان بمنع الانتهاء والتسليم الذري المقفل.
- **شروط القبول قبل التطبيق:** مراجعة توقيعات RPCs ونتائج Advisors وخطة التراجع، ثم موافقة صريحة منفصلة. لا Data Migration.
- **أثر الإنتاج:** لا يوجد حاليًا.

## المرحلة 5: Invitation Flow

- **الهدف:** دعوة بريدية يراجع المستخدم صلاحياتها قبل الإرسال ويحدد كلمته بنفسه.
- **الجداول:** `user_invitations`، `invitation_access_grants`.
- **الشاشات:** إنشاء دعوة، مراجعة، حالة الدعوات.
- **المخاطر:** كشف `service_role`، إعادة استخدام الدعوة، فشل جزئي بعد إرسال البريد.
- **شروط القبول:** Admin API خلف خدمة خلفية آمنة فقط، لا `service_role` أو أسرار في React، انتهاء/إلغاء، Idempotency، Audit.
- **أثر الإنتاج:** محدود؛ يطلق أولًا في بيئة اختبار ولا يستبدل إنشاء المستخدم الحالي.

## المرحلة 6: Central Access Grants

- **الهدف:** مصدر مركزي للدور داخل الموديل والجهة والنطاق والوقت.
- **الجداول:** `platform_role_assignments`، `permission_definitions`، `module_role_templates`، `module_role_template_permissions`، `user_module_access`.
- **الشاشات:** عرض Grants وقوالب الأدوار دون تفعيل على الموديلات.
- **المخاطر:** تصعيد صلاحية، أكثر من دور، Role Template من موديل آخر.
- **شروط القبول:** القيود الفريدة، RLS، RPC ذرية، `lock_version`، اختبارات NULL وانتهاء الوقت.
- **أثر الإنتاج:** Shadow data فقط؛ القرار القديم نافذ.

## المرحلة 7: Admin UI

- **الهدف:** لوحة الإدارة والمستخدمون والموديلات والجهات والصلاحيات.
- **الجداول:** قراءة جداول المراحل السابقة.
- **الشاشات:** Dashboard، Users، Invitations، Modules، Organizations.
- **المخاطر:** الاعتماد على إخفاء الأزرار أو عرض بيانات حساسة.
- **شروط القبول:** رفض الخادم لكل مسار غير مصرح، RTL وجوال، حالات خطأ واضحة، لا كلمات مرور أو أسرار.
- **أثر الإنتاج:** Feature Flag لـ`system_owner` فقط.

## المرحلة 8: Permission Requests

- **الهدف:** مدير الموديل يطلب ومالك النظام يقرر وينفذ.
- **الجداول:** `permission_change_requests`، `permission_rejection_reasons`.
- **الشاشات:** قوائم الحالات وتفاصيل المقارنة والتنفيذ.
- **المخاطر:** تنفيذ مزدوج أو اعتماد المدير لطلبه.
- **شروط القبول:** دورة الحالات المقيدة، رفض بسبب إلزامي، التنفيذ Idempotent، Audit ذري.
- **أثر الإنتاج:** لا يغير Grant إلا بعد اعتماد وتنفيذ system_owner.

## المرحلة 9: Temporary Permissions

- **الهدف:** صلاحيات مجدولة ومنتهية تلقائيًا وتنبيه مسبق.
- **الجداول:** `access_expiration_notifications`، واستخدام تواريخ Grants.
- **الشاشات:** أقرب انتهاء وإدارة التنبيهات.
- **المخاطر:** الاعتماد الأمني على Cron أو اختلاف المنطقة الزمنية.
- **شروط القبول:** RLS يمنع بعد `ends_at` مباشرة، المهمة Idempotent، التواريخ UTC والعرض Asia/Riyadh.
- **أثر الإنتاج:** يطبق أولًا على Grants تجريبية.

## المرحلة 10: Assignment Alerts

- **الهدف:** رصد الأعمال المفتوحة للمستخدم المعطل وإعادة إسنادها، مع بقاء الإسناد اليومي لمدير الموديل وتدخل `system_owner` عبر جميع الموديلات عند الحاجة.
- **الجداول:** `assignment_alerts`، Adapters قراءة لكل موديل.
- **الشاشات:** قائمة إعادة الإسناد وتفاصيل المورد.
- **المخاطر:** مرجع عام غير صحيح أو إعادة إسناد عبر الموديلات.
- **شروط القبول:** تحقق المورد داخل RPC الموديل، مدير الموديل محصور في موديله وجهاته، `system_owner` مخول مركزيًا، البديل لديه Grant فعال، والسبب وAudit إلزاميان.
- **أثر الإنتاج:** قراءة أولًا؛ لا تنفيذ جماعي.

## المرحلة 11: Audit Dashboard

- **الهدف:** بحث مركزي في الدعوات والمنح والتعطيل والإسناد.
- **الجداول:** `audit_logs`، والقرار المعتمد لـ`audit_log_contexts` أو الحقول الموسعة.
- **الشاشات:** سجل التدقيق والفلاتر والتفاصيل.
- **المخاطر:** بطء JSONB أو كشف قيم حساسة.
- **شروط القبول:** فهارس مناسبة، Redaction، لا UPDATE/DELETE، صلاحيات system_owner/auditor فقط.
- **أثر الإنتاج:** قراءة فقط.

## المرحلة 12: ربط الرقابة المالية

- **الهدف:** أول موديل يستخدم Central Grants تدريجيًا.
- **الجداول:** Backfill مرشح من `financial_control_members` إلى `user_module_access` دون حذف المصدر.
- **الشاشات:** مقارنة العضوية ومنح الرقابة المالية.
- **المخاطر:** تعدد الأدوار القديم وMapping `specialist` والجهات.
- **شروط القبول:** Compare Mode بلا فروقات حرجة، اختبار 33 ملاحظة، Feature Flag وتراجع موثق.
- **أثر الإنتاج:** تدريجي لمجموعة مستخدمين؛ Legacy متاح للتراجع.

## المرحلة 13: ربط الركائز

- **الهدف:** نقل قرار الوصول للركائز بعد استقرار الرقابة المالية.
- **الجداول:** Backfill من `workspace_members` وقوالب الركائز.
- **الشاشات:** مقارنة العضوية والمنح.
- **المخاطر:** التأثير على دوال `private` الحالية أو دورة الاعتماد.
- **شروط القبول:** ظهور 7/22/33/73، جميع الأدوار والانتقالات تعمل، Compare Mode مستقر.
- **أثر الإنتاج:** تدريجي مع تراجع Legacy.

## المرحلة 14: اختبار الأمان

- **الهدف:** إثبات العزل ومنع التصعيد والتجاوز.
- **الجداول:** لا جداول جديدة إلا إصلاحات معتمدة منفصلة.
- **الشاشات:** جميع شاشات الإدارة والموديلات.
- **المخاطر:** BOLA/IDOR، JWT قديم، NULL، سباق `lock_version`، RPC عامة.
- **شروط القبول:** اختبارات RLS إيجابية وسلبية، لا anon Grants، Advisors، اختبار تعطيل فوري، مراجعة SECURITY DEFINER.
- **أثر الإنتاج:** اختبارات Staging ثم فحوص قراءة في الإنتاج.

## المرحلة 15: الإطلاق المحدود

- **الهدف:** تشغيل الإدارة المركزية لمجموعة صغيرة مع مراقبة.
- **الجداول:** لا Schema جديد أثناء نافذة الإطلاق.
- **الشاشات:** Dashboard ومراقبة Audit والتنبيهات.
- **المخاطر:** فقد الوصول أو حمل دعم مرتفع.
- **شروط القبول:** حساب `system_owner` الجديد مستقل ومفعل بـMFA، Runbook، مالك احتياطي، استرداد آمن، مؤشرات نجاح، قناة Incident، خطة تراجع مجربة.
- **أثر الإنتاج:** نعم، محدود وتدريجي.

## بوابات الجودة المشتركة

لكل مرحلة تنفيذية:

1. مراجعة الملفات التي ستتغير قبل التنفيذ.
2. Migration واحدة صغيرة قابلة للمراجعة بعد اعتمادها فقط.
3. اختبار معاملة Rollback وفحص ساكن.
4. RLS مفعلة ومفروضة، ولا Grants لـ`anon`.
5. اختبارات دور مسموح ودور ممنوع وWorkspace/Organization أخرى.
6. اختبار `lock_version` وتزامن عمليتين.
7. Audit للقيم السابقة والجديدة والسبب.
8. `npm run build` واختبار الكمبيوتر والجوال.
9. توثيق التراجع قبل التطبيق.
10. توقف للمراجعة قبل Commit/Push أو تطبيق المرحلة التالية.

## مؤشرات نجاح البرنامج

- صفر تعديل صلاحية خارج طلب/دعوة موثق.
- صفر دخول لموديل معطل من مستخدم تشغيلي.
- الصلاحيات المنتهية تمنع عند أول طلب بعد الانتهاء.
- جميع المستخدمين المعطلين ذوي الأعمال المفتوحة لديهم Alerts.
- جميع تغييرات الدعوات والمنح والتعطيل والإسناد لها Audit مترابط.
- لا فروقات غير مفسرة بين Legacy وCentral في نهاية Compare Mode.
