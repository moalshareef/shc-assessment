# خطة الانتقال إلى الصلاحيات المركزية

## 1. الهدف

نقل إدارة الوصول تدريجيًا إلى نموذج مركزي دون تعطيل الرقابة المالية أو الركائز، ودون تغيير صلاحيات المستخدم الحالي أثناء التصميم. لا تتضمن هذه الوثيقة SQL قابلًا للتطبيق.

## 2. خط الأساس الحالي

- `profiles.role` دور عام قديم بقيم محدودة، و`profiles.is_active` متاح.
- `workspace_members` يربط المستخدم بالـWorkspace بدور واحد من `owner/manager/member/viewer`.
- `financial_control_members` يسمح حاليًا بعدة أدوار للمستخدم في مساحة الرقابة المالية.
- جداول الرقابة المالية تعتمد `workspace_id` وعضوية متخصصة وRLS/RPCs قائمة.
- الركائز تعتمد عضوية Workspace ودوالها الحالية داخل `private`.
- `audit_logs` هو السجل المركزي الحالي.

## 3. استراتيجية الانتقال

### 3.1 لا Big Bang

- لا يزال القرار القديم هو المصدر أثناء إنشاء الجداول والواجهة المركزية.
- لا تحذف ولا تعدل العضويات القديمة في المراحل الأولى.
- تبنى طبقة توافق تقرأ النموذجين وتقارن النتيجة.
- ينتقل موديل واحد في كل مرة خلف Feature Flag قابل للتراجع.

### 3.2 Compatibility Layer

عقد طبقة التوافق المعتمد:

```text
canAccessModule(user, module, organization, permission, record)
  ├── central grant decision
  ├── legacy adapter decision
  └── mode: legacy | compare | central
```

- `legacy`: القرار القديم فقط.
- `compare`: القرار القديم نافذ، والفروقات تسجل دون منع.
- `central`: القرار المركزي نافذ مع إبقاء القديم للتراجع المؤقت.

### 3.3 Mapping الأدوار

يعتمد Mapping التالي كأساس للانتقال، لكنه لا يطبق آليًا قبل التحقق من كل مستخدم وجهة ودور:

| المصدر | القالب المركزي المقترح | ملاحظة |
|---|---|---|
| `workspace_members.owner` | قالب مدير الموديل أو Owner خاص بالموديل | لا يتحول إلى `system_owner`. |
| `workspace_members.manager` | `module_manager` | حسب الموديل والجهة. |
| `workspace_members.member` | `employee` | يحتاج جهة ونطاقًا. |
| `workspace_members.viewer` | `viewer` | نطاق القراءة يحدد صراحة. |
| `financial_control_members.owner` | دور إشراف الرقابة المالية | لا يمنح دعوات مركزية. |
| `financial_control_members.manager` | مدير الرقابة المالية | وفق المسار المعتمد. |
| `financial_control_members.action_owner` | موظف تنفيذ | `assigned_records` غالبًا. |
| `financial_control_members.viewer` | Viewer الرقابة المالية | قراءة حسب النطاق. |
| `financial_control_members.specialist` | يحتاج قرار Mapping | دور قديم لا يستخدم في المسار المبسط الجديد، ولا يحذف تلقائيًا. |

## 4. إنشاء أول `system_owner` بأمان

1. ينشأ حساب Auth إداري جديد ومستقل عن الحساب التشغيلي الحالي، ويثبت امتلاكه عبر قناة إدارية آمنة دون بريد hard-coded في الكود.
2. لا يرقى الحساب التشغيلي الحالي ولا تتغير صلاحياته عند إنشاء الحساب الجديد.
3. يحدد مالك المشروع UUID للحساب الإداري الجديد بعد التحقق منه، ثم تنفذ عملية Bootstrap لمرة واحدة في معاملة مضبوطة بعد اعتماد Migration.
4. تتحقق العملية من عدم وجود `system_owner` فعال قبل الإدخال.
5. تسجل هوية منفذ التهيئة وسببها في Audit.
6. يختبر الدخول وMFA واسترداد الحساب.
7. تزال أو تعطل آلية Bootstrap بعد النجاح؛ لا تبقى RPC عامة.

لا يمنح `profiles.role='system_admin'` تلقائيًا دور `system_owner`.

## 5. دورة الدعوة

### قبل الإرسال

- إنشاء `user_invitations` كـ`draft`.
- حفظ الجهات وصلاحيات الدعوة بصورة normalized.
- التحقق من البريد المكرر والدور الواحد لكل موديل وجهة والتواريخ.

### عند الإرسال

- خدمة خلفية آمنة وموثوقة فقط تستخدم Supabase Auth Admin.
- لا يستدعي React Auth Admin ولا يحتوي `service_role` أو Secret.
- تحديث الدعوة إلى `sent` وAudit في تدفق موثوق قابل للتعويض إذا فشل البريد.

### عند القبول

- ربط `auth.users.id` و`profiles.id`.
- إنشاء `user_organizations` و`user_module_access` ذريًا.
- إذا فشل Grant واحد تبقى الدعوة `accepted` مع حالة معالجة واضحة، ولا يصبح المستخدم `active` جزئيًا.

## 6. تعطيل المستخدم

1. قفل Profile وقراءة Grants والإسنادات المفتوحة.
2. تعيين `profiles.is_active=false` وتسجيل السبب.
3. إبطال الجلسات/Refresh Tokens أو استخدام آلية تعليق Auth المعتمدة من خدمة موثوقة.
4. إلغاء فاعلية Grants دون حذفها.
5. إنشاء `assignment_alerts` للإسنادات المفتوحة بواسطة Adapters الموديلات.
6. كتابة Audit مترابط لكل العملية.

لا تغير سجلات `created_by/updated_by/assigned_by` التاريخية، ولا تحذف مستخدم Auth.

## 7. تعطيل الموديل

- تحديث `platform_modules.status=disabled` بسبب إلزامي.
- منع مساره تشغيليًا عبر RLS/قرار الوصول، لا Router فقط.
- لا تحذف Workspace أو Grants أو بيانات الموديل.
- يبقى مرئيًا لـ`system_owner` ولـ`auditor` وفق القراءة.
- إعادة التفعيل عملية مستقلة بعد اختبارات سلامة.

## 8. الرقابة المالية كموديل أول

### التحضير

- إنشاء سجل Module مرتبط بـWorkspace `financial-control`.
- تعريف قوالب الرقابة المالية المعتمدة.
- ربط الجهات الرسمية بـ`organizations` بعد مراجعة Mapping `departments`.
- تحويل العضويات إلى Grants مرشحة دون تفعيلها.

### المقارنة

- تشغيل `compare` لكل قرار قراءة وإجراء على عينة مستخدمين.
- مقارنة الدور والنطاق والتاريخ والجهة والنتيجة.
- معالجة حالات تعدد الأدوار الحالية يدويًا؛ النموذج الجديد يسمح بدور واحد لكل موديل وجهة.

### التحويل

- تفعيل `central` لمجموعة صغيرة.
- اختبار 33 ملاحظة ودورة الموظف والمدير والمستندات المرجعية.
- مراقبة 7 أيام عمل على الأقل دون فروقات حرجة.
- توسيع التفعيل تدريجيًا.

### الإسناد أثناء الانتقال

- يبقى مدير الرقابة المالية مسؤولًا عن الإسناد اليومي داخل الموديل.
- يختبر `system_owner` صلاحية الإسناد وإعادة الإسناد الشاملة في حالات تعطيل المستخدم والتصعيد فقط.
- لا تستخدم لوحة الإدارة تحديثًا مباشرًا لجداول الرقابة المالية؛ تمر العملية عبر RPC الموديل وتكتب Audit.

### التراجع

- إعادة Feature Flag إلى `legacy`.
- لا Rollback للبيانات التشغيلية؛ Grants المركزية تبقى معلمة وغير نافذة للتحليل.

## 9. الركائز بعد الرقابة المالية

- لا تعدل دوال `private` أو سياسات الركائز خلال ربط الرقابة المالية.
- تنشأ قوالب مستقلة لدورة الركائز.
- تعاد اختبارات ظهور 7 ركائز و22 فرعية و33 سؤالًا و73 معيارًا.
- يستخدم نفس نمط `legacy/compare/central`.

## 10. الصلاحيات المؤقتة

- عند Backfill تحفظ البداية الحالية والنهاية من العضوية القديمة إن وجدت.
- الغياب لا يعني صلاحية غير محدودة تلقائيًا إلا بعد اعتماد Mapping.
- RLS يفحص الوقت مباشرة.
- التنبيه المجدول وظيفة تشغيلية منفصلة لا يعتمد عليها المنع الأمني.

## 11. Audit أثناء الانتقال

- لا ينشأ سجل تدقيق مكرر.
- كل Grant منقول يحمل `source='manual_migration'` ومعرف Batch.
- يسجل Audit القيم القديمة وMapping الجديد والجهة والدور والنطاق.
- تقارير المطابقة لا تكتب بيانات حساسة أو Tokens.

## 12. اختبارات الانتقال

- مستخدم بلا Profile يمنع.
- مستخدم معطل يمنع فورًا رغم JWT قديم.
- Grant مستقبلي أو منتهٍ يمنع.
- موديل `draft/disabled` يمنع تشغيليًا.
- مستخدم جهة أخرى لا يرى السجل.
- `assigned_records` لا يوسع إلى سجلات الجهة.
- مدير موديل لا يرسل دعوة ولا ينفذ طلبه.
- `platform_admin` لا يعدل `system_owner`.
- اختلاف القرار القديم والجديد يسجل في Compare Mode.
- التراجع إلى Legacy يعيد الوصول السابق دون تغيير بيانات الأعمال.

## 13. موانع الانتقال

- عدم اعتماد ERD وقاموس Permissions.
- عدم وجود Mapping موثق للجهات والأدوار المتعددة.
- عدم توفر قناة Auth Admin خلفية ومحمية.
- عدم اعتماد سياسة تعليق الجلسات وMFA والاسترداد.
- وجود فروقات صلاحية غير مفسرة في Compare Mode.
- عدم اكتمال اختبارات RLS السلبية ومتعدد الجهات.

## 14. القرارات الانتقالية المعتمدة

- إنشاء `organizations` مع Mapping متحقق منه إلى `departments`، دون استبدال فوري.
- إبقاء `workspaces` مصدر الهوية التشغيلية وإضافة `platform_modules` للحوكمة.
- اعتماد Compatibility Layer ونقل موديل واحد في كل مرة.
- اعتماد حساب `system_owner` جديدًا ومستقلًا عن الحساب التشغيلي الحالي.
- اعتماد خدمة خلفية آمنة للدعوات ومنع `service_role` في React.
- الإسناد اليومي لمدير الموديل، والتدخل المركزي عند الحاجة لـ`system_owner` عبر جميع الموديلات.

## 15. ما لا يحدث في هذه المرحلة

- لا تعديل لقاعدة البيانات أو Supabase.
- لا إنشاء Migration أو Seed.
- لا تعديل `profiles/workspaces/workspace_members/financial_control_members`.
- لا تغيير لصلاحيات المستخدم الحالي.
- لا تعطيل موديل أو مستخدم.
- لا إرسال دعوة فعلية.
