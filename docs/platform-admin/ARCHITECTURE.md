# معمارية إدارة المنصة والصلاحيات المركزية

## 1. الهدف والنطاق

تصميم وحدة مؤسسية مستقلة لإدارة المنصة والموديلات والجهات والمستخدمين والدعوات والصلاحيات المؤقتة والطلبات والإسنادات والتدقيق. هذه الوثيقة تصميم فقط؛ لا تعتمد أي Migration ولا تغير سلوك الرقابة المالية أو ركائز كفاءة الإنفاق.

## 2. المبادئ الحاكمة

- إدارة المنصة منفصلة عن بيانات الأعمال داخل كل موديل.
- `system_owner` هو أعلى دور تشغيلي، لكنه ليس PostgreSQL superuser ولا يملك `service_role` في المتصفح.
- لا يوجد دخول نيابة عن المستخدم أو انتحال جلساته.
- الدعوة البريدية تنفذ من خدمة خلفية موثوقة فقط، ولا تخزن كلمات المرور في التطبيق.
- الدور النهائي يحدد قبل إرسال الدعوة، ثم يحول إلى منح وصول بعد قبولها.
- لكل مستخدم دور واحد فقط لكل `(module, organization)`، باستثناء `system_owner` الذي يملك وصولًا منصيًا عامًا محكومًا.
- انتهاء الصلاحية يطبق عند كل قرار وصول، ولا يعتمد فقط على مهمة دورية تغير الحالة.
- التعطيل بديل الحذف للمستخدم والموديل والعضوية.
- كل عملية حساسة ذرية، تتحقق من `lock_version`، وتكتب في `audit_logs`.

القرارات المعمارية والفنية الآمنة الواردة في هذه الحزمة معتمدة كأساس للتنفيذ المرحلي. أي تغيير لاحق عليها يحتاج موافقة صريحة، أما القيم التشغيلية القابلة للتهيئة مثل مدد التنبيه وانتهاء الدعوة فتحدد قبل مرحلة التنفيذ المعنية.

## 3. المكونات المعمارية

### 3.1 Platform Administration Module

واجهة وخدمات إدارية منفصلة بمسار مستقل، لا تستورد خدمات الموديلات التشغيلية ولا تقرأ جداول أعمالها إلا عبر عقود إشراف محددة. تشمل لوحة المؤشرات والمستخدمين والدعوات والموديلات والجهات وطلبات الصلاحيات والتدقيق وإعادة الإسناد.

### 3.2 Module Registry

اعتمد إبقاء `workspaces` أساسًا تشغيليًا للموديلات الحالية والجديدة، وإنشاء `platform_modules` كطبقة حوكمة مركزية فوقه. يحتفظ سجل الحوكمة بحالة `draft/active/disabled` والمسار ونسخة الإعداد ووقت التفعيل والتعطيل، ويرتبط بالـWorkspace بعلاقة واحد إلى واحد دون تكرار الهوية التشغيلية.

الموديل الجديد يمر بالمسار:

`draft → تطوير واختبار → active → disabled`

التعطيل لا يحذف Workspace أو بيانات الموديل، ويمنع الوصول التشغيلي مع إبقاء القراءة الإدارية لـ`system_owner`.

### 3.3 User and Invitation Management

- يسجل طلب الدعوة والصلاحيات المقترحة قبل الإرسال.
- يرسل رابط الدعوة عبر Supabase Auth Admin من خدمة خلفية آمنة فقط، مثل Edge Function محمية أو Backend مؤسسي.
- يمنع استدعاء Auth Admin من React، ولا يظهر مفتاح `service_role` أو أي Secret في حزمة المتصفح.
- بعد قبول الدعوة ينشأ أو يربط `profiles`، ثم تتحول صلاحيات الدعوة إلى منح وصول فعلية ذريًا.
- الدعوة أحادية الاستخدام، لها رمز/معرف Auth غير قابل لإعادة الاستخدام، وتاريخ انتهاء وحالات واضحة.

### 3.4 Organization Management

اعتمد إنشاء `organizations` كسجل جهات هرمي يدعم الجهة الرئيسية والفرعية، مع ربط انتقالي آمن وموثق بـ`departments`. يرتبط المستخدم بعدة جهات مع جهة رئيسية واحدة. تستمر `departments` الحالية كمرجع تشغيلي قديم أثناء الانتقال، ولا ينشأ Mapping واحد إلى واحد إلا بعد التحقق من أن السجلين يمثلان الجهة نفسها.

### 3.5 Role Templates

قوالب أدوار خاصة بكل موديل. يعرّف القالب مجموعة الإجراءات، ولا يمنح الوصول وحده. المنح تربط المستخدم بالموديل والجهة والقالب ونطاق البيانات. التخصيص الاستثنائي مؤجل إلى Overrides صريحة ومدققة، ولا يعدل القالب المشترك.

### 3.6 Module and Organization Access Grants

المصدر المركزي المستقبلي للقرار هو `user_module_access`. كل Grant يحدد:

- المستخدم.
- الموديل.
- الجهة.
- قالب الدور.
- نطاق البيانات.
- البداية والنهاية.
- الحالة وسبب الإلغاء.

القيد الفريد يمنع أكثر من Grant حالي لنفس المستخدم والموديل والجهة.

### 3.7 Temporary Access

الصلاحية المؤقتة تحفظ بـ`starts_at/ends_at`. الوصول الفعلي يتطلب:

`status = active AND starts_at <= now() AND (ends_at IS NULL OR ends_at > now())`

وبذلك تنتهي الصلاحية أمنيًا فورًا حتى لو تأخرت مهمة التنبيه. مهمة مجدولة ترسل تنبيهًا مسبقًا وتحدّث الحالة التشغيلية إلى `expired` بصورة قابلة لإعادة التشغيل.

### 3.8 Permission Change Requests

مدير الموديل لا ينشئ مستخدمًا ولا يرسل دعوة ولا يمنح صلاحية. ينشئ طلبًا يمر عبر:

`new → under_review → approved أو rejected → executed`

`system_owner` وحده يقرر وينفذ المنح النهائية. سبب الرفض إلزامي ويختار من قائمة جاهزة مع ملاحظة قصيرة اختيارية.

### 3.9 Assignment Oversight

الإسناد اليومي داخل كل موديل مسؤولية `module_manager` ويمارس فقط ضمن الموديل والجهات والصلاحيات الممنوحة له. يملك `system_owner` صلاحية الإسناد وإعادة الإسناد في جميع الموديلات للتدخل عند الحاجة، مثل تعطيل مستخدم أو غياب مدير الموديل أو معالجة تعارض أو تصعيد مركزي، دون الدخول نيابة عن أي مستخدم.

عند تعطيل مستخدم تبقى سجلاته التاريخية باسمه. تنشأ تنبيهات لإسناداته المفتوحة حسب Adapter خاص بكل موديل. لا تحدث وحدة الإدارة جداول الموديلات مباشرة؛ تستدعي RPC إسناد أو إعادة إسناد خاصة بالموديل، تتحقق من صلاحية المنفذ وتسجل القديم والجديد والسبب.

### 3.10 Audit and Security Monitoring

يستمر `audit_logs` كسجل مركزي. العمليات الإدارية تكتب القيم السابقة والجديدة والمستخدم والوقت والسبب ومعرف الارتباط داخل معاملة واحدة. تقرأ لوحة التدقيق السياق من الحقول المهيكلة المقترحة أو من بيانات Audit المتوافقة مؤقتًا.

## 4. حدود الطبقات

```text
React Admin UI
      │
      ▼
Public Admin RPC / trusted server endpoints
      │
      ├── Platform policy checks
      ├── Invitation orchestration
      ├── Access decision engine
      └── Audit writer
              │
              ▼
Platform tables ── Compatibility adapters ── Existing module memberships
                                              ├── workspace_members
                                              └── financial_control_members

Module UIs ── Module services/RPCs ── Module business tables
```

- واجهة الإدارة لا تتصل بـAuth Admin مباشرة.
- محرك القرار المركزي لا يمنح إذن إجراء داخل موديل إلا بعد التحقق من الدور والنطاق وحالة المستخدم والموديل والجهة والتواريخ.
- RLS هي خط الدفاع النهائي، وإخفاء الأزرار تحسين تجربة فقط.

## 5. استمرار الرقابة المالية والركائز دون تأثر

1. تبقى `workspace_members` و`financial_control_members` وسياسات RLS الحالية كما هي في مراحل التصميم والبناء الأولية.
2. تكتب طبقة التوافق منح الوصول المركزية بالتوازي مع العضويات القديمة بعد اعتماد مرحلة الربط، دون جعلها مصدر القرار فورًا.
3. تستخدم اختبارات مقارنة للتأكد أن القرار المركزي والقرار القديم يعطيان النتيجة نفسها.
4. تربط الرقابة المالية أولًا خلف Feature Flag لمستخدمين محددين.
5. لا ينتقل مصدر الصلاحية إلى المركز إلا بعد تحقق الأمان والتراجع.
6. تربط الركائز بعد استقرار الرقابة المالية، مع عدم تعديل دوالها الحالية قبل ذلك.
7. تعطيل موديل جديد لا يؤثر على Workspace آخر أو جداول مشتركة.

## 6. تدفق دعوة المستخدم

1. `system_owner` يدخل البريد والاسم والجهات والمنح المقترحة.
2. النظام يتحقق من عدم وجود دعوة فعالة أو مستخدم قائم بالبريد.
3. تحفظ الدعوة كـ`draft` وتعرض شاشة مراجعة.
4. RPC موثوق يقفل الدعوة ويتحقق من `lock_version` والمنح.
5. خدمة خلفية تستدعي `inviteUserByEmail` وتحدث الحالة إلى `sent` وتكتب Audit.
6. المستخدم يقبل الدعوة ويحدد كلمة المرور بنفسه.
7. Callback موثوق يربط `auth.users.id` بـ`profiles.id` ويحوّل المنح المقترحة إلى Grants.
8. تصبح الدعوة `accepted` ثم يصبح المستخدم `active` بعد نجاح إنشاء جميع المنح.

## 7. دورات الحياة المعتمدة

| الكيان | المسار |
|---|---|
| الدعوة | `draft → sent → accepted → active`، مع `expired/cancelled` |
| المستخدم | `invited → active → suspended`؛ التعليق لا يحذف Profile أو الأعمال التاريخية |
| الموديل | `draft → active → disabled` |
| طلب الصلاحية | `new → under_review → approved أو rejected → executed` |
| Grant | `scheduled → active → expired`، أو `revoked` بقرار وسبب |

ينشأ حساب `system_owner` جديد ومستقل عن الحساب التشغيلي الحالي، ومخصص لإدارة المنصة. لا يرفع الحساب الحالي تلقائيًا إلى هذا الدور، ولا يستخدم حساب المالك كهوية موظف لتنفيذ أعمال الموديلات أو للدخول نيابة عن مستخدم آخر.

## 8. قرار الوصول

تسلسل القرار الملزم:

1. جلسة Auth صالحة وغير ملغاة.
2. `profiles.is_active = true` وعدم وجود تعليق أمني فعال.
3. الموديل `active`، أو المستخدم `system_owner` في وضع الإدارة.
4. Grant فعال زمنيًا للموديل والجهة، ما لم يكن `system_owner`.
5. قالب الدور يسمح بالإجراء.
6. نطاق البيانات يسمح بالسجل المطلوب.
7. سياسة RLS أو RPC تعيد التحقق من جميع الشروط داخل قاعدة البيانات.

## 9. القرارات المعتمدة للتنفيذ المرحلي

- `platform_role_assignments` هو سجل أدوار المنصة، ولا توسع قيمة `profiles.role` لهذا الغرض.
- `organizations` هو المرجع المؤسسي المستقبلي مع Mapping انتقالي متحقق منه إلى `departments`.
- `workspaces` أساس التشغيل و`platform_modules` طبقة الحوكمة المركزية.
- الدعوات من خدمة خلفية آمنة فقط، مع SMTP مؤسسي عند الإطلاق، ومنع `service_role` داخل React.
- حساب `system_owner` جديد ومستقل، مع MFA إلزامي قبل الإنتاج وآلية استرداد محكومة.
- قالب دور منصي فعال واحد للمستخدم، وامتداد `organization_records` لا يشمل الجهات الفرعية إلا بعلم صريح.
- التخصيص الاستثنائي مؤقت ومدقق ولا يمنح Permissions المحجوزة.
- `audit_log_contexts` هو الامتداد المعتمد لفهرسة سياق Audit مع بقاء `audit_logs` السجل المركزي.
- مدد الدعوة والتنبيهات قيم إعداد قابلة للتهيئة تعتمد قبل تشغيل كل ميزة ولا تخزن ثابتة في الواجهة.
