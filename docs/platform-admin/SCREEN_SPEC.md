# مواصفات شاشات لوحة إدارة المنصة

## 1. الإطار العام

- مساحة إدارة مستقلة بعنوان «إدارة المنصة» لا تظهر إلا بحسب دور المنصة.
- Header يوضح أن المستخدم في نطاق الإدارة لا في موديل تشغيلي.
- Sidebar: لوحة التحكم، المستخدمون، الدعوات، الموديلات، الجهات، طلبات الصلاحيات، إعادة الإسناد، سجل التدقيق.
- كل شاشة تدعم حالات التحميل والخطأ وعدم وجود بيانات، ولا تحول خطأ RLS إلى حالة فارغة.
- العمليات الحساسة تعرض مراجعة أثر قبل التنفيذ، وتستخدم `lock_version` ورسائل تعارض واضحة.

## 2. لوحة التحكم الرئيسية

### المؤشرات

- المستخدمون النشطون.
- المستخدمون المعطلون.
- الدعوات المعلقة والمنتهية.
- الموديلات النشطة والمسودة والمعطلة.
- الصلاحيات التي تنتهي خلال 30/14/7 أيام.
- الأعمال المفتوحة التابعة لمستخدمين معطلين.
- طلبات الصلاحيات الجديدة وتحت المراجعة.

### الأقسام

- «يتطلب إجراء»: دعوات فاشلة، صلاحيات منتهية قريبًا، طلبات جديدة، إسنادات معلقة.
- «آخر أحداث التدقيق»: المنفذ، الحدث، المتأثر، الموديل، الجهة، الوقت.
- «صحة الوصول»: Grants غير متوافقة مع العضوية، موديل معطل وله نشاط، صلاحية انتهت ولم تعالج إشعاراتها.

لا تعرض الصفحة بيانات أعمال الموديلات التفصيلية؛ تعرض أعدادًا وروابط إشراف فقط.

## 3. المستخدمون

### القائمة

- بحث بالاسم والبريد والمسمى.
- تصفية بالحالة والجهة والموديل والدور والنطاق ونوع الصلاحية.
- أعمدة: المستخدم، الحالة، الجهة الرئيسية، عدد الجهات، عدد الموديلات، أقرب انتهاء، أعمال تحتاج إعادة إسناد.

### تفاصيل المستخدم

- الهوية الأساسية وحالة Auth وحالة Profile دون عرض أسرار.
- الجهات مع تمييز الجهة الرئيسية.
- جدول Grants: الموديل، الجهة، الدور، النطاق، البداية، النهاية، الحالة.
- سجل الدعوات والطلبات والتغييرات.
- الأعمال المفتوحة التي تحتاج إعادة إسناد عند التعطيل.
- أزرار `system_owner`: تعديل الصلاحيات، تعطيل/إعادة تفعيل، إنشاء إعادة إسناد.
- لا يظهر زر «دخول كمستخدم» مطلقًا.

### التعطيل

قبل التنفيذ تعرض الشاشة:

1. عدد الجلسات التي ستبطل.
2. عدد Grants التي ستتوقف.
3. عدد الأعمال المفتوحة المتوقع تحويلها إلى Alerts.
4. سبب إلزامي.

النتيجة تبقي كل الأعمال التاريخية باسم المستخدم.

## 4. دعوة مستخدم

نموذج متعدد الخطوات:

1. **الهوية:** البريد، الاسم.
2. **الجهات:** جهة أو أكثر، جهة رئيسية واحدة إلزامية.
3. **الصلاحيات:** الموديل، الجهة، قالب الدور، نطاق الوصول، بداية ونهاية.
4. **المراجعة:** ملخص يمنع الإرسال عند التعارض أو التكرار.
5. **الإرسال:** `system_owner` فقط؛ يظهر تاريخ الانتهاء وحالة التسليم.

قواعد الواجهة:

- لا يوجد حقل كلمة مرور.
- زر الإرسال يستدعي خدمة خلفية آمنة فقط؛ لا تستدعي الشاشة Supabase Auth Admin مباشرة ولا تحمل `service_role`.
- لا يسمح بدورين داخل نفس الموديل والجهة.
- نهاية الصلاحية بعد البداية.
- لا يرسل البريد قبل حفظ كل Grants المقترحة بنجاح.
- إعادة الإرسال عملية مستقلة ومدققة، ولا تعيد استخدام دعوة مستهلكة.

## 5. الموديلات

### القائمة

- الرمز والاسم من `workspaces`.
- الحالة من `platform_modules`: `draft/active/disabled`.
- المسار، عدد الأدوار، الجهات، المستخدمون، تاريخ التفعيل، آخر تعديل.

### التفاصيل

- تعريف الموديل والمسار والإعدادات غير السرية.
- قوالب الأدوار وPermissions لكل قالب.
- الجهات المرتبطة والمستخدمون حسب الجهة.
- سجل التغييرات واختبارات التفعيل.
- أزرار إنشاء مسودة، إرسال للاختبار، تفعيل، تعطيل.

### التفعيل والتعطيل

- التفعيل يتطلب شروط قبول مكتملة ولا ينفذه إلا `system_owner`.
- التعطيل يتطلب سببًا ويعرض أثره، ولا يحذف البيانات.
- الموديل المعطل يظهر فقط لـ`system_owner` و`auditor` وفق سياسة القراءة.

## 6. الجهات

- عرض شجري للجهة الرئيسية والفرعية.
- بحث وتصفية بالحالة والنوع.
- تفاصيل الجهة: المستخدمون، الجهة الرئيسية لكل مستخدم، الموديلات، Grants، Mapping إلى `departments`.
- إنشاء وتعديل وتعطيل دون حذف.
- منع نقل جهة تحت أحد أبنائها.
- شاشة مقارنة قبل تغيير الأب توضح أثر نطاق `organization_records`.

## 7. طلبات الصلاحيات

### القوائم

تبويبات: جديد، تحت المراجعة، معتمد، مرفوض، منفذ.

### التفاصيل

- مقدم الطلب وموديله وجهته.
- المستخدم أو البريد المستهدف.
- الصلاحية الحالية والمطلوبة جنبًا إلى جنب.
- المبرر وتاريخ الصلاحية.
- سجل الحالة والتعليقات.

### الإجراءات

- مدير الموديل: إنشاء الطلب وتعديله في `new` وإرساله للمراجعة.
- `system_owner`: بدء المراجعة، اعتماد، رفض، تنفيذ.
- الرفض: سبب جاهز إلزامي وملاحظة اختيارية.
- التنفيذ: من نفس الشاشة بعد الاعتماد، ويعرض Grant الناتج ولا يقبل التنفيذ مرتين.

## 8. إعادة الإسناد

- قوائم حسب الموديل والمستخدم المعطل والجهة وعمر التنبيه.
- تفاصيل المورد دون نسخ بيانات حساسة إلى جدول Alerts.
- اختيار الموظف البديل من Grants الفعالة المناسبة فقط.
- سبب إعادة الإسناد إلزامي.
- التنفيذ عبر RPC الموديل، ثم تحديث Alert إلى `reassigned` وعرض Audit.
- دعم التجميع للعرض فقط؛ التنفيذ الجماعي يؤجل حتى اعتماد ضوابطه.

### توزيع مسؤولية الإسناد

- مدير الموديل يرى زر «إسناد» و«إعادة إسناد» داخل موديله وجهاته فقط، وهو المسؤول عن العمل اليومي.
- `system_owner` يرى شاشة إشراف شاملة لكل الموديلات، ويتدخل عند تعطيل مستخدم أو غياب المدير أو وجود تصعيد أو تعارض يتطلب حوكمة مركزية.
- كل تدخل مركزي يعرض سبب التدخل والموديل والجهة والمسؤول السابق والجديد قبل التنفيذ.
- لا يمنح الإسناد `system_owner` جلسة المستخدم أو يسمح له بتنفيذ العمل باسمه.

## 9. سجل التدقيق

### الحقول

- من قام بالإجراء.
- نوع عملية الصف والحدث الوظيفي.
- المستخدم المتأثر.
- الموديل والجهة.
- الجدول والسجل.
- القيم السابقة والجديدة.
- السبب ومعرف الطلب/الارتباط.
- الوقت.

### البحث والتصفية

- فترة زمنية.
- المنفذ والمتأثر.
- الموديل والجهة.
- نوع الحدث والجدول.
- Correlation ID.

القراءة فقط؛ لا تعديل ولا حذف. إظهار JSON الخام مقيد بـ`system_owner/auditor` مع إخفاء الحقول الحساسة.

## 10. حالات العرض ورسائل الأمان

- `403`: «لا تملك الصلاحية لهذا الإجراء» دون كشف وجود سجل خارج النطاق.
- `40001`: «عُدّل السجل من مستخدم آخر؛ حدّث الصفحة».
- صلاحية منتهية: توضح تاريخ الانتهاء ولا تعرض زر تنفيذ.
- مستخدم معطل: Badge ثابت وأزرار الأعمال مخفية/معطلة مع تفسير.
- موديل معطل: يمنع فتح المسار التشغيلي ويعرضه إداريًا فقط.
- دعوة منتهية أو مستهلكة: لا يعاد استخدامها؛ ينشأ إرسال جديد وفق السياسة.

## 11. الاستجابة وإتاحة الاستخدام

- على الكمبيوتر: جداول مع لوحة تفاصيل جانبية أو صفحة مستقلة.
- على الجوال: بطاقات مختصرة وفلاتر في Drawer، ولا تعتمد العمليات الحساسة على Hover.
- جميع الحقول Labels صريحة ورسائل الأخطاء مرتبطة بها.
- التواريخ تعرض بالعربية مع الاحتفاظ بقيمة ISO تقنيًا.
- حالات اللون مدعومة بنص وأيقونة، لا باللون وحده.

## 12. شروط قبول واجهة الإدارة

- لا يظهر أي إجراء غير مسموح، ومع ذلك يرفضه الخادم عند استدعائه يدويًا.
- لا توجد كلمة مرور أو مفتاح إداري في الواجهة.
- لا يوجد دخول نيابة عن المستخدم.
- حساب `system_owner` المعروض في الإدارة مستقل عن الحساب التشغيلي الحالي، ولا توجد عملية واجهة لترقية الحساب الحالي تلقائيًا.
- كل عملية تغير حالة تعرض نتيجة Audit ومعرفها.
- التعطيل لا يحذف سجلًا.
- اختبارات الكمبيوتر والجوال والاتجاه RTL مكتملة.
